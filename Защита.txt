1. ЧТО ВООБЩЕ ДЕЛАЕТ ПРОГРАММА (суть в одном абзаце)
--------------------------------------------------------------------------------
Консольное приложение шифрует и расшифровывает файлы и директории симметричным
алгоритмом AES-256-GCM. Ключ не берётся напрямую из пароля, а выводится из пароля
и случайной соли функцией PBKDF2-HMAC-SHA256. Шифрование потоковое (блоками по
4 КБ), запись — атомарная (через QSaveFile), целостность проверяется тегом
аутентификации GCM. То есть это пример аутентифицированного шифрования (AEAD),
а не просто шифрования.


2. КРИПТОГРАФИЧЕСКАЯ ТЕОРИЯ — это спросят в первую очередь
--------------------------------------------------------------------------------

Почему симметричное шифрование, а не асимметричное:
  Шифруем большие файлы → нужна скорость. Симметричные шифры (AES) на порядки
  быстрее асимметричных (RSA). Один и тот же ключ используется и для шифрования,
  и для дешифрования.

Что такое AES-256-GCM:
  - AES — блочный шифр (Advanced Encryption Standard), блок 128 бит.
  - 256 — длина ключа в битах (32 байта → KEY_SIZE = 32).
  - GCM (Galois/Counter Mode) — режим работы блочного шифра. Делает две вещи
    одновременно:
      1) Шифрует данные (конфиденциальность) — работает как потоковый шифр на
         основе режима счётчика (CTR).
      2) Аутентифицирует — вычисляет тег (16 байт), который гарантирует, что
         данные не изменили и пароль верный. Это и есть AEAD — Authenticated
         Encryption with Associated Data.
  Главное преимущество GCM: при дешифровании, если хоть один бит шифротекста,
  тега, nonce или ключа неверен — EVP_DecryptFinal_ex вернёт ошибку. То есть
  неверный пароль обнаруживается автоматически.

Что такое nonce и зачем он:
  Nonce (number used once) — 12 байт случайных данных (NONCE_SIZE = 12). В GCM
  nonce задаёт начальное состояние счётчика. Критично: одна и та же пара
  (ключ, nonce) НИКОГДА не должна повторяться, иначе GCM ломается полностью.
  У нас он генерируется случайно при каждом шифровании через RAND_bytes.
  12 байт — рекомендованный стандартом размер для GCM.

Что такое соль (salt) и зачем PBKDF2:
  Если из пароля делать ключ напрямую (например, просто SHA256(пароль)), то:
    - одинаковые пароли → одинаковые ключи;
    - можно заранее посчитать таблицы (rainbow tables).
  Соль — 16 случайных байт (SALT_SIZE = 16), уникальна для каждого файла.
  PBKDF2 (Password-Based Key Derivation Function 2) берёт пароль + соль и
  прогоняет их через HMAC-SHA256 100 000 раз (PBKDF2_ITERATIONS). Это:
    - замедляет перебор паролей (brute-force) в 100 000 раз;
    - делает ключ уникальным даже для одинаковых паролей (за счёт соли).


3. ФОРМАТ ЗАШИФРОВАННОГО ФАЙЛА — обязательно знать наизусть
--------------------------------------------------------------------------------

  MAGIC(7) | VERSION(1) | SALT(16) | NONCE(12) |   DATA   | TAG(16)
  └──────── заголовок HEADER_SIZE = 36 байт ────────┘

  Поле        Размер    Зачем
  ----------  --------  --------------------------------------------------------
  MAGIC="ENCFILE"  7    Сигнатура — отличить «наш» файл от обычного
  VERSION=1        1    Версия формата (на будущее, для совместимости)
  SALT            16    Чтобы при дешифровании заново вывести тот же ключ
  NONCE           12    Чтобы при дешифровании восстановить состояние GCM
  DATA       переменный Зашифрованные байты (= размеру исходного, без padding)
  TAG             16    Тег аутентификации, проверяется в конце дешифрования

  HEADER_SIZE = MAGIC_SIZE + FORMAT_VERSION_SIZE + SALT_SIZE + NONCE_SIZE
              = 7 + 1 + 16 + 12 = 36

  Важный момент: соль и nonce хранятся в файле в ОТКРЫТОМ виде. Это нормально и
  безопасно — они не секретны по определению. Секретен только пароль.


4. АРХИТЕКТУРА И ПАТТЕРНЫ (вопросы по ООП)
--------------------------------------------------------------------------------

CryptoManager — паттерн Singleton (Одиночка):
    static CryptoManager& instance() {
        static CryptoManager instance;   // создаётся один раз
        return instance;
    }
  - Конструктор приватный, копирование и присваивание удалены (= delete) —
    нельзя создать второй экземпляр.
  - Используется «Майерсовский синглтон» — static локальная переменная,
    потокобезопасна в C++11+.
  - Зачем синглтон тут: менеджер шифрования — это единая точка доступа к
    операциям, состояния он не хранит, дублировать его нет смысла.

ITest — паттерн Strategy / интерфейс:
  Чисто абстрактный класс (интерфейс) с двумя чисто виртуальными методами name()
  и run(). Идея — каждый тест реализует этот интерфейс, а TestMenu хранит
  коллекцию ITest и запускает их единообразно. Это даёт расширяемость: добавить
  новый тест = добавить класс, не меняя меню.

Разделение ответственности (SRP):
  - CryptoManager — только криптооперации над файлами/директориями.
  - PasswordKeyDerivation — только работа с паролем и выводом ключа.
  - crypto_constants.h — только константы (никакой логики).
  - TestMenu / TestUtils / ITest — инфраструктура тестирования.
  - main.cpp — только пользовательский интерфейс (меню).


5. РАЗБОР ЛОГИКИ ШИФРОВАНИЯ ПО ШАГАМ (encryptFile)
--------------------------------------------------------------------------------
  1) Проверки: файл существует и ещё не зашифрован — чтобы не зашифровать дважды.
  2) Генерация соли RAND_bytes(salt, 16) — криптостойкий генератор OpenSSL.
  3) Вывод ключа из пароля и соли через PBKDF2.
  4) Генерация nonce RAND_bytes(nonce, 12).
  5) Инициализация контекста OpenSSL: EVP_CIPHER_CTX_new → EVP_EncryptInit_ex с
     EVP_aes_256_gcm(), затем установка ключа и nonce.
  6) Открытие QSaveFile — пишем во временный файл, оригинал не трогаем.
  7) Запись заголовка: MAGIC, VERSION, SALT, NONCE.
  8) Потоковое шифрование: читаем по 4096 байт, EVP_EncryptUpdate шифрует блок,
     сразу пишем результат. Так можно шифровать файлы любого размера, не
     загружая их целиком в память.
  9) Финализация EVP_EncryptFinal_ex.
 10) Получение тега EVP_CIPHER_CTX_ctrl(..., EVP_CTRL_GCM_GET_TAG, ...) и запись
     его в конец файла.
 11) Очистка: OPENSSL_cleanse(key) затирает ключ в памяти нулями.
 12) outFile.commit() — атомарно подменяет оригинал зашифрованной версией.

Дешифрование (decryptFile) — зеркально:
  1) Читает и проверяет MAGIC — если не совпадает, файл не наш.
  2) Проверяет VERSION.
  3) Читает SALT и NONCE из заголовка.
  4) Вычисляет размер данных: dataSize = размер_файла − HEADER_SIZE − TAG_SIZE.
  5) Выводит ключ из пароля и ПРОЧИТАННОЙ из файла соли.
  6) Потоково дешифрует DATA.
  7) Читает TAG, устанавливает его (EVP_CTRL_GCM_SET_TAG).
  8) EVP_DecryptFinal_ex проверяет тег. Если result != 1 → неверный пароль или
     повреждение → cancelWriting(), оригинал не пострадал.


6. ОБРАБОТКА ДИРЕКТОРИЙ
--------------------------------------------------------------------------------
  encryptDirectory / decryptDirectory используют QDirIterator с флагом
  Subdirectories — рекурсивный обход всех вложенных файлов. Для шифрования
  пропускаются уже зашифрованные файлы (!isFileEncrypted), для дешифрования —
  наоборот, берутся только зашифрованные. Если один файл упал — success = false,
  но остальные продолжают обрабатываться.


7. МЕРЫ БЕЗОПАСНОСТИ (это любят спрашивать)
--------------------------------------------------------------------------------
  1) PBKDF2 со 100 000 итераций — защита от перебора пароля.
  2) Уникальная соль на каждый файл — защита от rainbow tables и одинаковых
     ключей.
  3) Уникальный nonce на каждый файл — обязательное условие безопасности GCM.
  4) AEAD (тег GCM) — защита целостности и подлинности, автоопределение
     неверного пароля.
  5) OPENSSL_cleanse — затирание ключа в памяти после использования (обычный
     memset компилятор может выкинуть как «лишний», OPENSSL_cleanse — нет).
  6) QSaveFile + commit() — атомарная запись: при сбое (питание, ошибка)
     оригинал не повреждается, временный файл просто удаляется.
  7) Очистка пароля в main.cpp через std::fill(...'\0') после каждой операции.
  8) Валидация пароля 8–256 символов.
  9) Валидация путей/прав перед операцией (isValidFileForEncryption и т.п.).


8. СБОРКА И ЗАВИСИМОСТИ
--------------------------------------------------------------------------------
  - CMake ≥ 3.5, стандарт C++17.
  - Qt5 Core — для работы с файлами (QFile, QSaveFile, QDir, QDirIterator,
    QString, QByteArray).
  - OpenSSL (SSL + Crypto) — вся криптография (AES-GCM, PBKDF2, RAND_bytes,
    OPENSSL_cleanse).
  - Компилятор — MinGW64 из MSYS2.


9. ВЕРОЯТНЫЕ ВОПРОСЫ ПРЕПОДАВАТЕЛЯ И ОТВЕТЫ
--------------------------------------------------------------------------------
В: Почему не зашифровать просто паролем как ключом?
О: Пароль обычно короткий и не равен 32 байтам, у него низкая энтропия. PBKDF2
   «растягивает» его до полноценного 256-битного ключа и замедляет перебор.

В: Можно ли расшифровать без пароля?
О: Нет. В файле хранятся только соль и nonce (они не секретны). Без пароля ключ
   вывести нельзя, а тег GCM не даст получить осмысленные данные.

В: Что будет при неверном пароле?
О: Из неверного пароля выведется неверный ключ, и EVP_DecryptFinal_ex вернёт
   ошибку проверки тега — программа сообщит «Неверный пароль или файл повреждён»
   и не повредит файл.

В: Зачем MAGIC и VERSION?
О: MAGIC — быстро понять, наш ли это файл (и не шифровать дважды). VERSION —
   задел на будущее: если формат изменится, старые файлы можно будет отличить и
   обработать корректно.

В: Почему шифрование потоковое?
О: Чтобы не загружать большой файл целиком в ОЗУ — читаем и шифруем блоками
   по 4 КБ.

В: Зачем QSaveFile, а не обычная запись?
О: Атомарность. Пишем во временный файл, и только commit() подменяет оригинал.
   При сбое исходный файл цел.

В: GCM меняет размер файла?
О: Сами данные — нет (GCM работает как потоковый шифр, без padding). Файл
   вырастает ровно на заголовок (36 байт) + тег (16 байт) = 52 байта.

#  ЧАСТЬ 2. АРГУМЕНТАЦИЯ АРХИТЕКТУРЫ ПРОЕКТА
################################################################################

Для защиты важно не просто описать структуру, а ОБОСНОВАТЬ каждое решение:
почему так, какие были альтернативы и почему их отвергли.


1. ОБЩИЙ ПРИНЦИП: слоистая архитектура с разделением ответственности
--------------------------------------------------------------------------------
Проект разбит на три логических слоя:

  ┌─────────────────────────────────────────┐
  │  Слой представления (UI)                 │  main.cpp
  │  меню, ввод/вывод, очистка пароля        │
  ├─────────────────────────────────────────┤
  │  Слой бизнес-логики (крипто)             │  CryptoManager
  │  шифрование файлов/директорий            │  PasswordKeyDerivation
  ├─────────────────────────────────────────┤
  │  Слой конфигурации/инфраструктуры        │  crypto_constants.h
  │  константы, тесты                        │  ITest/TestMenu/TestUtils
  └─────────────────────────────────────────┘

Аргумент: каждый слой зависит только от нижележащего и не знает о вышележащем.
CryptoManager ничего не знает про меню — его можно встроить в GUI, веб-сервис
или вызвать из тестов без единого изменения. Это прямое следование принципу
разделения ответственности (SRP из SOLID): у каждого модуля одна причина для
изменения.

Доказательство на практике: режим --test переиспользует тот же CryptoManager
через тот же публичный интерфейс. Если бы логика шифрования была вшита в
обработчик меню, тесты были бы невозможны без дублирования кода.


2. ПОЧЕМУ CryptoManager — Singleton
--------------------------------------------------------------------------------
Решение: приватный конструктор, удалённые копирование и присваивание,
единственный экземпляр через static.

Аргументы за:
  1) Объект не имеет изменяемого состояния — все данные (соль, nonce, ключ)
     живут только внутри методов как локальные переменные. Раз состояния нет,
     два экземпляра неотличимы — значит, второй не нужен.
  2) Единая точка доступа к криптооперациям из любого места (меню, тесты) без
     передачи объекта через параметры.
  3) Майерсовский синглтон (static локальная переменная) — потокобезопасен по
     стандарту C++11, ленивая инициализация, корректное уничтожение.

Честное замечание для защиты (покажет глубину понимания): синглтон здесь близок
к набору статических методов. Альтернатива — сделать все методы static и не
плодить экземпляр вообще. Singleton выбран потому, что он ОСТАВЛЯЕТ возможность
в будущем добавить состояние (например, кэш ключей, счётчик операций, настройки)
без переписывания всех вызовов. То есть это задел на расширяемость, а не догма.

Почему не отдельные свободные функции? Тогда нет единого пространства имён,
методы валидации (isValidFileForEncryption) и приватные хелперы (encryptFile,
processFile) «растеклись» бы по файлу. Класс инкапсулирует публичный API и
прячет реализацию (private-методы) — это инкапсуляция.


3. ПОЧЕМУ PasswordKeyDerivation вынесен в отдельный класс
--------------------------------------------------------------------------------
Аргумент: вывод ключа из пароля — это отдельная зона ответственности с
собственной криптологикой (PBKDF2, выбор соли, число итераций) и собственными
правилами валидации (длина 8–256).

Если бы это лежало внутри CryptoManager, нарушился бы SRP: у менеджера появилось
бы две причины меняться — изменение алгоритма шифрования И изменение схемы
вывода ключа. Вынеся KDF, мы можем:
  - заменить PBKDF2 на Argon2/scrypt, не трогая CryptoManager;
  - переиспользовать вывод ключа в любых других сценариях;
  - тестировать валидацию пароля изолированно.

Методы сделаны static, потому что функция вывода ключа чистая — результат
зависит только от входов (пароль + соль), внутреннего состояния нет. Класс здесь
играет роль модуля-группировщика связанных функций.


4. ПОЧЕМУ КОНСТАНТЫ ВЫНЕСЕНЫ В crypto_constants.h
--------------------------------------------------------------------------------
Аргумент: все магические числа (размеры ключа, соли, nonce, тега, число
итераций, сигнатура) собраны в одном месте.

  1) Единый источник истины (DRY). HEADER_SIZE вычисляется из других констант —
     измени SALT_SIZE, и заголовок пересчитается автоматически везде. Нет риска,
     что в одном файле 16, а в другом 12.
  2) Связность формата. Шифратор и дешифратор обязаны использовать ОДНИ И ТЕ ЖЕ
     размеры, иначе формат «развалится». Общий заголовок это гарантирует на
     уровне компиляции.
  3) constexpr — вычисление на этапе компиляции, ноль накладных расходов в
     рантайме, плюс проверка типов.


5. ПОЧЕМУ ТЕСТИРОВАНИЕ ПОСТРОЕНО НА ИНТЕРФЕЙСЕ ITest
--------------------------------------------------------------------------------
Решение: ITest — чистый абстрактный класс, TestMenu хранит
vector<unique_ptr<ITest>> и запускает тесты единообразно.

Аргумент — это паттерн Strategy + принцип открытости/закрытости (OCP):
  - Добавление нового теста = создание нового класса, реализующего ITest. Меню
    (TestMenu) при этом не меняется — оно работает с абстракцией, а не с
    конкретными тестами.
  - TestMenu зависит от ИНТЕРФЕЙСА, а не от реализаций — это инверсия
    зависимостей (DIP). Конкретные тесты подставляются «снаружи».
  - unique_ptr даёт чёткое владение и автоматическое освобождение — это
    композиция (а не зависимость): меню владеет жизненным циклом тестов.

Почему именно так, а не if/else по номеру теста? Жёсткий switch пришлось бы
править при каждом новом тесте — нарушение OCP, рост связанности. Полиморфизм
через ITest убирает эту связь.


6. ПОЧЕМУ ВЫБРАНЫ Qt + OpenSSL, а не одно из двух
--------------------------------------------------------------------------------
Аргумент — разделение «что» и «чем»:
  - OpenSSL отвечает за криптографию — это промышленный, проверенный аудитом
    стандарт. Писать свой AES или PBKDF2 — грубейшая ошибка («не изобретай свою
    криптографию»).
  - Qt отвечает за работу с файловой системой. QSaveFile даёт атомарную запись
    из коробки, QDirIterator — рекурсивный обход. На голом C++ это пришлось бы
    писать вручную и кросс-платформенно.

То есть каждая библиотека закрывает ту зону, где она сильна. Связь с ними —
зависимость (используем API), а не наследование, что снижает связанность:
теоретически файловый слой можно заменить на std::filesystem, не трогая крипто.


7. СВОДНАЯ ТАБЛИЦА АРХИТЕКТУРНЫХ РЕШЕНИЙ
--------------------------------------------------------------------------------
  Решение                    | Альтернатива            | Почему выбрано это
  ---------------------------+-------------------------+------------------------
  CryptoManager как Singleton| Статические функции /   | Нет состояния → один
                             | обычный класс           | экземпляр; задел на
                             |                         | расширение; единая
                             |                         | точка доступа
  ---------------------------+-------------------------+------------------------
  Отдельный                  | KDF внутри              | SRP: смена алгоритма
  PasswordKeyDerivation      | CryptoManager           | KDF не задевает
                             |                         | шифрование
  ---------------------------+-------------------------+------------------------
  Константы в общем заголовке| Числа по месту          | DRY + гарантия единого
                             | использования           | формата у шифратора и
                             |                         | дешифратора

ГЛАВНЫЙ ТЕЗИС ДЛЯ ЗАЩИТЫ
--------------------------------------------------------------------------------
Архитектура подчинена ДВУМ ЦЕЛЯМ: безопасность и сопровождаемость. Безопасность
диктует выбор проверенных примитивов (OpenSSL, AEAD, PBKDF2, атомарная запись),
а сопровождаемость — разбиение на слабосвязанные модули с единственной
ответственностью, что подтверждается возможностью переиспользовать криптослой в
тестовом режиме без изменений.
